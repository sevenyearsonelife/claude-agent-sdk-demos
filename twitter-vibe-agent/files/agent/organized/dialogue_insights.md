# 用户理解与困惑

## 已理解

### 技术概念
- Claude Code 的核心架构：nO 主循环引擎、h2A 异步消息队列、wU2 压缩器
- 单线程设计 vs 多智能体蜂群的权衡
- 上下文管理策略（三层存储、92% 压缩触发）
- 工具设计哲学（Regex vs 向量数据库、Markdown vs 数据库）
- 子代理管理机制（受控并行、深度限制）
- diff-centric 编辑理念
- 沙箱安全实现（OS 级 vs 应用层）

### 架构对比
- Claude Code vs Codex CLI 的核心差异
- 主动扫描 vs 懒加载的上下文策略
- 显式规划 vs 隐式规划的工作流
- 结构化工具 vs Shell 优先的设计哲学

### 实践经验
- CLAUDE.md 文件的配置模式（架构 72.6%、开发指南 44.8%）
- 328 个项目的实证研究数据
- "架构 + 依赖 + 项目概述"是最常见组合（21.6%）

## 困惑点

### 未在对话中明确体现

- 用户似乎已经对技术细节有较好理解

### 可能的深入方向

- 面试官视角：需要考察候选人对 LLM Agent 运行原理的理解
- 技术深度：中高级到资深工程师

### 岗位类型
- AI 工程师 / LLM 应用开发工程师
- Agent 系统架构师

## 难度偏好

### 层级分布
- 基础：核心概念、工作机制（如 ReAct 循环、压缩阈值）
- 进阶：设计权衡、架构对比（如单线程 vs 多智能体、Regex vs 向量数据库）
- 专家：底层实现、优化策略、安全机制

### 题目类型
- 概念解释题：考察核心组件作用
- 原理分析题：考察设计决策背后的原因
- 对比分析题：考察不同方案的权衡
- 场景应用题：考察实际问题的解决思路

## 风格偏好

### 口吻
- 面试官视角：引导候选人深入思考
- 技术严谨：确保术语准确、逻辑清晰
- 探索式：通过追问挖掘真实理解深度

### 语气强度
- 严谨专业：确保技术准确性
- 友好引导：鼓励候选人表达完整思路
- 犀利追问：对模糊回答要求澄清

### 题目特征
- 层层递进：从基础概念到深入原理
- 场景化：结合实际工程场景
- 对比式：通过对比不同方案考察理解深度

## 禁用项/避免

### 内容方面
- 避免纯定义背诵题（如"什么是XXX"）
- 避免脱离实际的空谈理论
- 避免过于宽泛无法深挖的话题

### 形式方面
- 避免完整改写内容，只输出结构化线索
- 避免混入个人理解和困惑
- 不生成完整的面试题文本，只提供可出题线索

### 技术边界
- 专注"Claude/LLM 运行原理"强相关内容
- 不偏离主题到通用编程或软技能
- 如果资料偏离主题，只提取相关部分
